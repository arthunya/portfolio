01
Why this dataset — decision rationale
Before writing a single line of SQL, pick the right data foundation
Dataset candidates considered
Three options were evaluated for the O2C reconciliation project:
✓ Chosen — Olist + synthetic settlement (hybrid)
Olist Brazilian E-Commerce (Kaggle, public domain) — 99,441 real orders, 7 relational tables, multi-payment-type data (credit card, boleto, voucher, debit card), timestamps covering purchase → approval → delivery. Used as the order/billing side.

A synthetic settlement file will be generated in L2 to simulate what a payment processor would return — matching the reality of enterprise O2C where order system and settlement system are always separate.

Why this over fully synthetic: Real data has real dirty data — nulls, timing anomalies, orphan records. That makes the reconciliation exercise meaningful, not just a clean demo.
✗ Not chosen — UCI Online Retail II
Invoice-level only. No payment or settlement records at all. Would require building both sides from scratch.
✗ Not chosen — Fully synthetic
Too much generation time. "Synthetic only" also carries less credibility in a portfolio context than real public data.
Download source
kaggle.com/datasets/olistbr/brazilian-ecommerce — free account required. Download all 8 CSV files. The olist_geolocation_dataset.csv (~1M rows) is not needed for reconciliation work — skip it or open in Python only, never Excel.
💡
Lesson: In a real BSA engagement, the dataset decision is a requirements conversation — what data does Finance currently export? What does the payment processor settlement file look like? The dataset choice shapes every query you write downstream.
02
Setup: DuckDB + DBeaver
Why not Excel, and how to get a SQL environment running in 10 minutes
Excel can open CSVs but fails at joining multiple tables and slows down past ~100K rows. SQL gives us reproducibility — every finding is a saved query anyone can re-run to verify.
Install DuckDB
# Terminal (Windows PowerShell or Mac/Linux)
pip install duckdb

# Then install DBeaver Community (free SQL editor UI)
# https://dbeaver.io/download/ → Community Edition
Connect DBeaver to DuckDB
DBeaver → New Connection → search "DuckDB" → set path to a new .duckdb file, e.g. C:/olist/olist.duckdb. DBeaver creates the file on first connect.
Load all CSVs as tables — run one at a time (Ctrl+Enter per block)
CREATE TABLE orders   AS SELECT * FROM read_csv_auto('H:/path/olist_orders_dataset.csv');
CREATE TABLE payments AS SELECT * FROM read_csv_auto('H:/path/olist_order_payments_dataset.csv');
CREATE TABLE items    AS SELECT * FROM read_csv_auto('H:/path/olist_order_items_dataset.csv');
⚠ Common error: path backslash
Windows paths use \ but SQL needs /. Change C:\folder\file.csv to C:/folder/file.csv.
⚠ Common error: running all queries at once
Highlight only the CREATE TABLE you want, then Ctrl+Enter. If you run all three at once, the second CREATE fires before the first finishes — you get "table does not exist" on the SELECT that follows.
💡
Why DuckDB instead of MySQL/PostgreSQL: DuckDB reads CSV directly with read_csv_auto() — no import wizard, no schema definition upfront. SQL syntax is standard and transfers to Snowflake, BigQuery, or Redshift with minimal changes.

Null analysis — orders table
Nulls in the wrong columns = data integrity exceptions
COUNT(column) counts only non-null rows. So COUNT(*) - COUNT(col) = number of nulls. This pattern replaces Excel's =COUNTBLANK() but works across 99K rows instantly.
SELECT
    COUNT(*)                              AS total_rows,
    COUNT(order_approved_at)              AS has_approved,
    COUNT(*) - COUNT(order_approved_at)   AS null_approved,
    COUNT(order_delivered_customer_date)  AS has_delivered,
    COUNT(*) - COUNT(order_delivered_customer_date) AS null_delivered
FROM orders;

Investigate — inspect the 14 records directly
Finding 14 is not enough — pull the actual rows before drawing any conclusions. This query retrieves every timeline-relevant column so we can see what the records look like.
-- pull the actual 14 records and inspect their timestamps
SELECT
    order_id,
    order_status,
    order_purchase_timestamp,
    order_approved_at,                  -- this column will be NULL for all 14 rows

FROM orders
WHERE order_approved_at IS NULL

The result is 14 rows. Scanning the order_purchase_timestamp column, all records appear to cluster in Jan–Feb 2017. That is an observation at this point, not a confirmed finding — the next query counts it precisely.
Confirm the pattern — count by month
STRFTIME trims a timestamp down to year-month only, then GROUP BY counts how many orders fall in each month. Excel equivalent: =TEXT(A2,"YYYY-MM") followed by a Pivot Table on that column.
-- trim timestamp to year-month, then count per month
SELECT
    STRFTIME(order_purchase_timestamp, '%Y-%m') AS purchase_month,
    COUNT(*) AS total
FROM orders
WHERE order_approved_at IS NULL
  AND order_status = 'delivered'
GROUP BY purchase_month
ORDER BY purchase_month;

Confirmed — all 14 records fall within just 2 months. A random error would scatter across the full 2-year dataset. Two months means a systemic event, not individual mistakes.
🔴 Finding — 14 delivered orders, no approval record
All 14 records cluster in Jan–Feb 2017 (2 in January, 12 in February). This is a systemic data gap, not random human error — likely a system outage or migration event in Q1 2017 that prevented approval timestamps from being recorded while fulfillment continued normally.

In a real Finance audit, these would be flagged as: revenue recognized with no corresponding approval event — audit trail incomplete for 14 transactions.
💡
SQL lesson — IS NULL vs = NULL: In SQL, NULL = NULL evaluates to unknown (not true), so WHERE col = NULL always returns zero rows. Always write IS NULL or IS NOT NULL.
📌
Note for later review: These two queries are investigative queries — written to confirm a hypothesis mid-analysis, not to produce a deliverable directly. Come back to these once STRFTIME, AND conditions, and GROUP BY feel fully familiar.

06
Amount mismatch — core reconciliation query
Do what customers paid match what the order system says they owe?
This is the core O2C reconciliation question: order total (items + freight) vs payment total. We use CTEs (WITH ... AS) to pre-aggregate each table to one row per order before joining — this prevents a critical bug (see Step 07).
WITH order_value AS (
    SELECT
        order_id,
        ROUND(SUM(price + freight_value), 2) AS order_total
    FROM items
    GROUP BY order_id
),
payment_value AS (
    SELECT
        order_id,
        ROUND(SUM(payment_value), 2)   AS payment_total,
        MAX(payment_installments)          AS installments
    FROM payments
    GROUP BY order_id
)
SELECT
    o.order_id,
    ov.order_total,
    pv.payment_total,
    ROUND(ov.order_total - pv.payment_total, 2) AS diff,
    pv.installments,
    CASE
        WHEN pv.installments > 1
         AND pv.payment_total > ov.order_total THEN 'INSTALLMENT_INTEREST'
        WHEN pv.payment_total < ov.order_total  THEN 'UNDERPAYMENT'
        ELSE 'OVERPAYMENT'
    END AS exception_type
FROM orders o
JOIN order_value   ov ON o.order_id = ov.order_id
JOIN payment_value pv ON o.order_id = pv.order_id
WHERE ABS(ov.order_total - pv.payment_total) > 0.01
ORDER BY ABS(diff) DESC;

INSTALLMENT_INTEREST (257): Expected business behavior — credit card installment fees in Brazil are charged on top of item price. Not a reconciliation error; exclude from exception report.

UNDERPAYMENT (83, R$199.52): Average R$2.40 per order — likely partial voucher redemption that wasn't fully offset. Flag for Finance review, low priority.

OVERPAYMENT (29, R$6.59 total): Average R$0.23 — floating-point rounding in payment processor. Below materiality threshold. Document and close.
07
Bug found: many-to-many JOIN fan-out
What went wrong the first time, and why CTEs fix it
The first version of the amount mismatch query joined items directly to payments without pre-aggregating. The result was wildly inflated numbers — one order showed a payment of R$109,312 against an order total of R$13,664.
🐛 Bug — many-to-many JOIN creates row multiplication
When an order has 3 items AND 2 payment rows, a direct JOIN creates 3×2 = 6 rows.
SUM(price) then counts each item price twice (once per payment row).
SUM(payment_value) counts each payment three times (once per item row).

Fix: Use CTEs to reduce each table to one row per order_id before joining. Then the join is always 1-to-1.
💡
Lesson: Always aggregate before joining when both sides of a JOIN can have multiple rows per key. This is called the "fan-out problem" and it's one of the most common silent data errors in SQL — the query runs without error, the numbers just look wrong. The fix is WITH table AS (SELECT key, SUM(...) ... GROUP BY key).

Canceled orders with payment — R$143,255
The biggest financial exposure in the dataset
625 orders have order_status = 'canceled' but still carry active payment records. This means the customer paid, the order was canceled, but there is no refund or write-off in the data.
SELECT
    COUNT(DISTINCT o.order_id) AS canceled_with_payment,
    ROUND(SUM(p.payment_value), 2) AS total_at_risk,
    ROUND(AVG(p.payment_value), 2) AS avg_per_order
FROM orders o
JOIN payments p ON o.order_id = p.order_id
WHERE o.order_status = 'canceled';

🔴 Critical — R$143,255 revenue exposure
Every single canceled order (625 of 625) has a payment record. This is the headline finding of the entire L0 layer. In a real Finance context, each of these 625 orders needs one of three dispositions: (1) refund issued, (2) write-off approved, or (3) payment reapplied to a replacement order. Problem statement (L0 deliverable)
Context
Olist marketplace processes 99,441 orders across four payment types over a 2-year period (2016–2018), totaling R$16,008,872.
Problem
Manual reconciliation between order records and payment settlements cannot detect exceptions at scale. SQL analysis identified R$143,255 of collected payments on 625 canceled orders with no refund or write-off record, 775 orders with missing item records, and a systemic data gap affecting 14 delivered orders in Q1 2017.
Impact
R$143,455 total material exposure. Manual reconciliation estimated at 2–3 business days per monthly close cycle. Exceptions discovered late increase refund processing cost and audit risk.
Objective
Design an automated O2C reconciliation system achieving ≥95% match rate on order-to-payment reconciliation, classifying exceptions by type and severity, and flagging material items for Finance review by T+1.
