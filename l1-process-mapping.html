01
The four stages every order goes through
Strip away the internal system jargon — this is what actually happens
Every online order — at Olist, or anywhere — moves through the same four stages, regardless of which team or system owns each one.
💡 The technical name for this kind of picture is a BPMN swimlane diagram — each stage sits in a lane owned by a different team or system. You don't need to remember the name, just the four boxes and the three flags. The full 8-step technical version (split by system event, for writing SQL exception queries) lives in the Lucidchart spec used for L2.
02
Where the money currently leaks
Straight from the L0 SQL findings — no sampling, no estimation
🔴 Biggest leak
625 orders were canceled, but the money was never given back. R$143,255 is sitting in limbo — nobody in the current process decided whether to refund it, write it off, or apply it to a replacement order.
⚠ Data gap
775 orders have no record of what actually shipped. If a customer disputes what they received, there's nothing in the system to check it against.
⚠ System event
14 orders were marked "delivered" without ever being formally approved. All 14 cluster in a two-month window (Jan–Feb 2017) — that pattern points to a system outage, not 14 separate human mistakes.
03
The fix — what changes
A computer checks every order; a person only reviews what doesn't add up
None of this needs a person checking every order. It needs a computer to check every order, and a person to look only at the ones that don't add up.
TodayAfter the fix
Someone finds problems only when they go looking, usually weeks laterProblems surface automatically the day they happen
Every order gets treated the same, whether it's fine or notOnly the ~0.7% of orders that don't add up need a human look
No consistent label for what kind of problem it isEvery flagged order comes pre-sorted: canceled-but-paid, missing shipment, underpaid, etc.
