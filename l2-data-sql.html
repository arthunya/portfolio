01
Query 1 — Auto match
Does what the customer paid match what the order system says they owe?
Pre-aggregate items and payments to one row per order_id using the CTE pattern from L0 (prevents the fan-out bug found there), then LEFT JOIN from orders so every order survives — even ones missing items or payments entirely.
CREATE OR REPLACE TABLE l2_auto_match AS
WITH order_value AS (
    SELECT order_id, ROUND(SUM(price + freight_value), 2) AS order_total
    FROM items GROUP BY order_id
),
payment_value AS (
    SELECT order_id, ROUND(SUM(payment_value), 2) AS payment_total,
           MAX(payment_installments) AS installments
    FROM payments GROUP BY order_id
)
SELECT
    o.order_id, o.order_status, o.order_approved_at,
    ov.order_total, pv.payment_total, pv.installments,
    ROUND(COALESCE(pv.payment_total, 0) - COALESCE(ov.order_total, 0), 2) AS diff,
    CASE WHEN ABS(COALESCE(pv.payment_total, 0) - COALESCE(ov.order_total, 0)) <= 0.01
         THEN 'MATCH' ELSE 'MISMATCH' END AS match_status
FROM orders o
LEFT JOIN order_value   ov ON o.order_id = ov.order_id
LEFT JOIN payment_value pv ON o.order_id = pv.order_id;

Why LEFT JOIN, not JOIN
JOIN (inner) silently drops any order with no matching item or payment row — the exact kind of gap that matters most in reconciliation. LEFT JOIN + COALESCE(x, 0) keeps every one of the 99,441 orders and turns "no data" into a real, calculable exception instead of a missing row.
💡
Trick: LEFT JOIN keeps every row from the main table even when the other side is empty. COALESCE fills that empty side with 0 so the math still works. The two always travel together in a reconciliation query.
02
Query 2 — Auto classify
Sort the 1,142 mismatches into the exception types Finance actually needs to act on
Built on top of l2_auto_match. A single CASE WHEN assigns exactly one exception_type per order across 6 categories plus a normal "OK" state.
CREATE OR REPLACE TABLE l2_exception_classified AS
SELECT *,
    CASE
        WHEN order_status = 'canceled' AND payment_total > 0 THEN 'CANCELED_WITH_PAYMENT'
        WHEN order_approved_at IS NULL AND order_status = 'delivered' THEN 'APPROVAL_GAP'
        WHEN order_total IS NULL THEN 'MISSING_ITEMS'
        WHEN match_status = 'MATCH' THEN 'OK'
        WHEN installments > 1 AND payment_total > order_total THEN 'INSTALLMENT_INTEREST'
        WHEN payment_total < order_total THEN 'UNDERPAYMENT'
        ELSE 'OVERPAYMENT'
    END AS exception_type
FROM l2_auto_match;
First attempt returned only 2 orders for CANCELED_WITH_PAYMENT — L0 had documented 625. See Step 03 for what went wrong and why.
03
Bug found: CASE WHEN priority order
CASE WHEN behaves like if/elif, not a set of independent ifs
Most canceled orders also have no item records at all — order_total is NULL. Checking MISSING_ITEMS before CANCELED_WITH_PAYMENT meant those orders were claimed by the first branch and never reached the canceled-order check.
🐛 Bug — wrong branch order in CASE WHEN
CASE WHEN stops at the first condition that evaluates true — it does not check every condition. Once a row matches order_total IS NULL, it is locked into MISSING_ITEMS even if it also satisfies order_status = 'canceled'.

Fix: reorder branches so business-status flags (canceled, approval gap) — which matter regardless of dollar amounts — are checked before amount-based flags. Result jumped from 2 → 622 orders correctly classified as CANCELED_WITH_PAYMENT.
💡
Lesson: CASE WHEN = if/elif, never if/if. Always order branches from most specific/business-critical to most general — the most important flag has to win the tie, not whichever condition happens to sit first in the pipeline logically.
04
Bug found: reusing "diff" for the wrong question
Not every exception type is answering the same question
Once CANCELED_WITH_PAYMENT was correctly classified, its reported exposure showed R$37,369.88 — far short of L0's R$143,255.60.
🐛 Bug — diff answers "reconciliation," not "risk exposure"
diff = payment_total − order_total, built to answer "does the amount match?" But for a canceled order, the real question is "how much money did we collect that we now need to refund or write off?" — that's payment_total on its own, regardless of whether it happens to equal order_total.

Fix: report SUM(payment_total) for CANCELED_WITH_PAYMENT instead of SUM(diff). Result matched L0 exactly: R$143,255.60.
💡
Lesson: The same column can't always answer two different business questions. Reconciliation exceptions ask "how far apart are the two numbers?" Risk-exposure flags ask "how much money is actually at stake?" Match the aggregate to the question being asked, not to whichever column is already sitting on the table.
05
Query 3 — Exception summary + severity
Roll every exception type up to HIGH / MED / LOW / EXP for Finance triage
Severity tier reuses L0's original priority table (Layer 0, Section 09). Financial exposure switches its aggregate per exception type per Step 04's fix.
CREATE OR REPLACE TABLE l2_exception_summary AS
SELECT
    exception_type,
    CASE exception_type
        WHEN 'CANCELED_WITH_PAYMENT' THEN 'HIGH'
        WHEN 'UNDERPAYMENT'          THEN 'MED'
        WHEN 'MISSING_ITEMS'         THEN 'MED'
        WHEN 'APPROVAL_GAP'          THEN 'MED'
        WHEN 'OVERPAYMENT'           THEN 'LOW'
        WHEN 'INSTALLMENT_INTEREST'  THEN 'EXP'
    END AS severity,
    COUNT(*) AS orders,
    ROUND(SUM(
        CASE WHEN exception_type = 'CANCELED_WITH_PAYMENT'
             THEN payment_total ELSE ABS(diff) END
    ), 2) AS financial_exposure
FROM l2_exception_classified
WHERE exception_type != 'OK'
GROUP BY exception_type;

06
Reconciling L2 against L0's original numbers
The counts shifted slightly — and that's a feature, not a discrepancy
L0's exception counts came from separate, independent ad-hoc queries — not a single unified classification. L2's job was to make every order resolve to exactly one exception type, which surfaced two refinements:
✓ CANCELED_WITH_PAYMENT: 622 vs 625
3 of L0's original 625 had a payment record present but valued at R$0.00 — no money actually changed hands. L2 filters on payment_total > 0, so those 3 are correctly excluded from the financial-risk count.
ℹ MISSING_ITEMS: 614 vs 775 · OVERPAYMENT: 30 vs 29
L0's separate queries weren't mutually exclusive — an order could be counted in more than one ad-hoc query at once. L2's single CASE WHEN tree assigns each order to exactly one bucket, so some orders that were double-counted in L0 now sit in a single, correct category. The full set still sums to exactly 99,441.
💡
Lesson: Independent exploratory queries (L0) are for finding patterns. A production classification (L2) has to be mutually exclusive and collectively exhaustive — every row gets exactly one label, and the labels sum to the total. Reconciling the two is itself part of the audit trail, not a sign something broke.
